* 2022~2024년 프롬프트엔지니어링
	* 역할부여, 단계별 사고, 예시 제공
	* 완벽한 이메일 한통을 쓰는 상황
	* 단발성 질의&응답에 유효
	* 장시간 복잡한 작업에서는 프롬프트 하나로는 한계
	* 뭘말할까?
* 2025년 컨텍스트엔지니어링
	* 적절한 맥락 정보를 체계적으로 제공
	* 관련문서, 대화이력, 도구정의 RAG 검색결과를 컨텍스트 윈도우에 넣음
	* 이메일에 첨부파일을 붙이는 상황
	* 뭘보여줄까? 뭘제공할까?
* 2026년 하네스엔지니어링
	* 제약조건 + 도구 + 피드백 루프 + 문서화 를 포함하는 전체 운영 환경
	* 이메일과 첨부파일을 넘어 업무환경을 전부 설계하는것
	* 어떤 환경에서 일하도록 할까?

---

* 컨텍스트는 희소 자원
	* 거대한 지시 파일이 실제 작업, 코드를 밀어냄
* 모든게 "중요"하면 아무것도 중요하지 않음
	* 에이전트가 대충 패턴 매칭만 수행
* 거대한 메뉴얼은 금방 부패
	* 옛날 규칙과 새 규칙이 뒤섞임
* 1000페이지의 메뉴얼이 아니라 지도를(주소제공)주도록

---

* 자기 결과물을 외부적으로 검증할 수 있도록한다
	* 자동 테스트 : 코드 -> 즉시 검증
	* 타입 체크 & 린터 : 규칙 위반 -> 즉각 피드백
	* 검증 다각화 : 코드 작성 모델 -> 검증모델
* 가이드 : 행동 전 방향 설정
* 센서 : 행동 후 결과 체크&교정

---
* 코드를 많이 생성하면 코드베이스에 무질서 축적
* 문서와 코드 불일치, 중복 코드, 미사용 파일
* 정리 전담 에이전트 : 사무실의 청소 담당자
	* 문서-코드 일치 검사 에이전트
	* 아키텍처 규칙 위반 탐지 에이전트
---

* 모델을 바꾸지 말고 환경을 바꾸라
* 실패에서 시작하라 : 실패시 구조적 방지 장치 추가
* 적게 넣어라 : 규칙을 너무 많이 넣으면 오히려 성능 저하, 비용 증가
* 말로 하지 말고 시스템으로 강제
	* 프롬프트로 하지말고
	* 린터, 테스트, CI 같은 도구들로 강제
